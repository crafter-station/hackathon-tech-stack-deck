## Root Cause Findings

### References placement

- `src/components/deck-shell.tsx` currently inspects only its immediate `children`.
- `src/app/[slug]/page.tsx` passes a single MDX component as the shell child.
- `References` is nested inside the rendered MDX tree, so the shell cannot actually discover it.
- Therefore the references area in the shell is effectively never being driven by real extracted data.

Recommended fix:

- Stop trying to infer references from rendered children.
- Add `references` as an explicit optional export in slide MDX files.
- Extend the generated slide manifest to include `references`.
- Render `slide.references` directly in `DeckShell`, in a dedicated stack directly above the footer.

### Iteration timeline mismatch

- The current `settings` branch in `src/components/slides/iteration-timeline.tsx` draws a radial icon that looks like a sun.
- The “now” row uses a grid of uniform cells, which makes the eye read two separate horizontal rows even if bars and icons are vertically swapped.

Recommended fix:

- Replace the settings icon with a true gear glyph.
- Change the timeline data structure so each now-step is rendered as a compact paired unit that clearly reads as “bar, then icon.”
- Keep the repeated sequence, but ensure each pair is visually grouped more strongly than the surrounding grid.

### Hydration error / nested `<p>`

- `mdx-components.tsx` globally remaps MDX paragraphs to a custom `Paragraph` component that renders a real `<p>`.
- Some slide files use explicit JSX `<p className="...">` wrappers with multiline text content.
- MDX then turns the inner text into a paragraph node too, which the custom `p` renderer wraps again.
- Result: `<p><p>...</p></p>`, which causes the hydration warning.

Confirmed affected/risky files:

- `src/content/slides/01-from-idea-to-internet.mdx`
- `src/content/slides/02-iteration-velocity.mdx`
- `src/content/slides/03-the-loop.mdx`
- `src/content/slides/06-move-fast-but-care.mdx`

Minimal safe fix:

- Replace those explicit wrapper `<p>` elements with `<div>`s carrying the same classes where appropriate.
- Do not change Quote semantics for this issue.
